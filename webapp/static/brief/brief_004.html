@DavidOndrej1 ♥426 · 2026-05-16 · 📹 26分钟演示视频
Agent 工具📹
「Hermes Agent just shipped /goal. And this might be the most important feature of 2026.」/goal 让 Hermes Agent 设定一个目标，用 judge 模型持续评估是否完成，没完成就自动继续下一轮，直到达成目标或 turn budget 用完。彻底解决了 Agent「做到一半就停」的问题。

Hermes Agent 文档描述（hermes-agent.nousresearch.com）：「After every turn a lightweight judge model checks whether the goal is satisfied by the assistant's last response. If not, Hermes automatically feeds a continuation prompt back into the same session and keeps working — until the goal is achieved, you pause or clear it, or the turn budget runs out.」

翻译：「每轮之后，一个轻量的 judge 模型检查目标是否已被满足。如果没有，Hermes 自动把一个续行提示注入同一会话并继续工作——直到目标达成、你暂停/清除它、或者 turn budget 用完。」

这与 steipete 在 5 月 11 日说的「All I want is codex automatically entering /review mode after it's done and just looping until it stops finding booboos」完全是同一个需求——他说要自己去建，Hermes 已经建好了。

@steipete · ♥1,490 + ♥3,645 · 2026-05-14

steipete 本期连发两个工具：① clawpatch 0.1.0（♥1,490）：把代码库分割成「语义功能单元（semantic feature slices）」，对每个单元进行 bug/质量/回归检查，把 Claude Code 的 harness 工程化标准化；② CodexBar 更新（♥3,645）：「The latest CodexBar update renders API costs wayyyy nicer」——把 Codex API 成本可视化成清晰的 dashboard。
clawpatch 的技术意义

clawpatch 解决了一个 Coding Agent 领域最实际的问题：你有一个 100,000 行的代码库，怎么让 Agent 系统性地找 bug，而不是随机瞎找？

clawpatch 的方法：把代码库按功能拆分成「语义单元」（比如「用户认证模块」「支付流程模块」），然后对每个单元单独运行 bug-finding Agent。这样既利用了 LLM 对小上下文窗口的最佳理解能力，又确保了全覆盖。steipete 说这就是 OpenClaw 内部用来「split all our projects into functional units to review and find bugs」的方法。

CodexBar 背后的故事

同日（♥3,645），steipete 分享了 CodexBar 的 API 成本可视化更新，引发了「People freaking out over my AI spend」的帖子（♥7,479）——他在 OpenAI 工作的 30 天内花了 .3M 的 API 费用（但作为 OpenAI 员工是公司支付的）。HN 帖子标题「OpenClaw Creator Spent .3M on OpenAI Tokens in 30 Days」，161 points，200+ 评论。这件事揭示了：当你把 AI 当成真正的「团队成员」来用时，API 成本是多大量级。

Coding Agent 架构
Hermes /goal + clawpatch：两个「让 Agent 真正完成工作」的工程解法合流
本期出现了两个方向相同但层级不同的工程创新：① Hermes /goal（持久目标 + judge 模型循环）解决了「Agent 做到一半就停」问题；② clawpatch（语义功能单元分割 + 逐单元 bug check）解决了「大代码库 Agent 化的上下文管理」问题。两者合在一起，是目前最接近「生产级 Coding Agent 工作流」的开源实现。Gary Marcus 的「神经符号主义胜利」论点在这里有了工程层面的体现：这些 harness 设计都是在用确定性的结构约束（judge 模型、功能单元分割）来控制不确定的 LLM 输出。

Hermes /goal 的技术实现细节

官方文档描述（hermes-agent.nousresearch.com）：在每轮对话结束后，一个「轻量 judge 模型」检查目标是否被满足。Judge 的判断标准「刻意保守」：只有当回应明确确认任务完成，才标记为完成（done）——这防止了 Agent「自我感觉良好但实际没完成」的问题。

这个设计的关键：judge 模型比主 Agent 便宜得多（可能是一个小型模型），所以每轮评估的成本可以控制。Turn budget（默认 20 轮）提供了自然的成本上限。

为什么这两个工具同时出现

这不是巧合——它反映了 Coding Agent 领域在 2026 年 Q2 进入「工程化成熟阶段」的共同认知：大家都意识到，让 Agent「更智能」是模型公司的事，让 Agent「可靠地完成任务」是 harness 工程师的事。这个认知的普及，正在驱动一批「Coding Agent Harness 工具」的涌现（clawpatch、watchmen、开源 Claude Code harness 等）。

「让 Agent 可靠地完成任务」是比「让 Agent 更智能」更近期可解决的工程问题。这个方向的工具将在 Agent 大规模企业部署阶段成为关键基础设施。

威胁 1 —— 成本问题的具体表现：Agent 工作流下，一个典型的企业 AI 任务可能消耗 100k-1M tokens（vs 聊天场景的 1k-10k）。Anthropic 的 /5 per million token 定价，在 Agent 规模下会变成每月数万美元的账单。这对 Claude Code 00/月订阅的用户（如 levelsio）形成瓶颈——他们要么换到 API 计费，要么换到竞争对手。

Your Data Agents Need Context — a16z · Jason Cui & Jennifer Li
a16z · 2026-03-10（本期引用）
a16z 基础设施团队定义了「AI 时代数据栈的新核心」：Context 层。核心论点：过去十年建好的现代数据栈（Snowflake/dbt）解决了「数据放哪里、怎么查」，但 Agent 无法有效使用这些数据——因为它不知道「revenue 字段指的是 GAAP 还是 bookings」。Context 层（语义定义 + 数据血缘 + 访问策略 + 知识图谱）是所有数据 Agent 的共同前提依赖。
▸三代数据栈：孤岛时代 → 现代数据栈（Snowflake/dbt）→ Agent-ready 语义层（正在发生）
▸a16z 原文：「Every company building a data agent will hit the same wall within six months: context.」
▸Context 层四个子组件：Semantic Layer（业务定义）· Lineage Graph（数据血缘）· Policy Layer（访问权限）· Knowledge Graph（实体关系）
▸投资方向：a16z 在寻找「从零为 Agent 设计的 Context Layer 产品」，而非传统数据目录改造版本

问题的本质（原文 + 翻译）

「Over the past year, the market has realized that data and analytics agents are essentially useless without the right context – they aren't able to tease apart vague questions, decipher business definitions, and reason across disparate data effectively.」

翻译：「过去一年，市场已经意识到，数据和分析 Agent 在没有正确上下文的情况下基本上是没用的——它们无法拆解模糊的问题、解读业务定义，也无法有效地跨不同数据进行推理。」

Context 层四个组件的技术细节

Semantic Layer（语义层）：不只是「revenue 是收入」，而是「本公司的 revenue 指的是 GAAP ASC 606 标准下的已确认收入，按季度统计，排除内部交易，以下情况视为 deferred revenue：SaaS 合同预付款、超过 12 个月的服务合同...」这种精度。

Lineage Graph（血缘图）：这个数字从哪来？经过了哪些变换？被哪些报表依赖？当 Agent 发现一个异常值时，它需要知道「这个数字是哪条 pipeline 生成的」才能判断是 bug 还是业务真实变化。

Policy Layer（权限策略层）：谁能看哪些数据？在什么条件下？传统 RBAC 不够用，需要「Agent 意图感知」的权限系统。

Knowledge Graph（知识图谱）：客户 → 合同 → 产品 → 订单 → 发票 → 收入。Agent 在回答「客户 A 的健康状况」时需要理解这条关系链。

a16z 的投资判断（原文）

「Every company building a data agent will hit the same wall within six months: context. The companies that solve this will be the picks and shovels of the agentic data economy. This is not a feature for a single product. It is infrastructure for an entire category.」
