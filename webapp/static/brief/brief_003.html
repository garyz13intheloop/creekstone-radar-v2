Microsoft MDASH
Microsoft Security · 2026-05-12 · @satyanadella ♥704
AI 安全 多 Agent
Multi-model Defense Agent System for Hardening。100+ 专业 Agent 协作互相验证，CyberGym 基准 88.45%（行业最高），实战发现 16 个 Windows 0-day，含 4 个 Critical RCE（内核 TCP/IP 栈和 IKEv2 服务）。
核心机制：多个 Agent 互相「争论」和验证漏洞，类似多人 code review 的完全自动化版本。每个 Agent 专注一个细分能力（语法分析、运行时行为、内存安全等），发现候选漏洞后交给「裁判 Agent」验证可利用性。

测试数据（来自 Microsoft 官方博客）：
• 私测集：植入的 21 个漏洞全部找到，零误报
• MSRC 历史案例回溯：clfs.sys 召回率 96%，tcpip.sys 100%
• 公开基准：CyberGym 1507 个真实漏洞，得分 88.45%（行业最高，超过 Anthropic Mythos）
• 实战成果：Patch Tuesday 发现 16 个 Windows 0-day，含 4 个 Critical RCE

Satya Nadella 原话：「AI vulnerability discovery has crossed from research curiosity to real-world capability. Our system argues with itself to find bugs no human reviewer caught.」

为什么重要：这是迄今最复杂的生产级多 Agent 系统之一的公开案例。「Agent 互相验证」的机制，对所有需要高可靠性的 Agent 系统都有参考价值，不只是安全领域。

Anthropic Claude Dreaming
Anthropic · Code with Claude 大会 · 2026-05-06
Agent 记忆 大厂
Agent 在会话间隙自动回顾过往任务、提炼经验写入记忆库，不修改模型权重。Harvey 开启 Dreaming 后，法律 Agent 任务完成率提升 6x。同期还发布：Outcomes（自评分循环）和 Multiagent Orchestration（主 Agent 分发子 Agent 并行）。
Dreaming 工作原理：不是模型训练，是「结构化笔记整理仪式」。后台定时进程读取 Agent 最近的会话记录，寻找三类模式：① 反复出现的失败（recurring mistakes）；② 跨任务收敛的工作流；③ 多个 Agent 共同呈现的偏好。然后重写记忆库：淘汰过时内容，提升核心内容权重。

Anthropic 类比：「类似海马体的记忆巩固——人类大脑在睡眠中回放白天事件，决定留什么、删什么。」

Harvey 案例：Harvey 的法律 Agent 此前反复在相同任务上失败，原因是「每次会话都忘了文件格式的特殊处理方式」。Dreaming 让这些 workaround 被记住，完成率从基线提升 6x。

三个功能对比：
• Dreaming（研究预览，需申请）：跨会话学习，自动写入记忆
• Outcomes（公开 Beta）：对照 rubric 自评分，让 Agent 知道哪里没做好
• Multiagent Orchestration（公开 Beta）：主 Agent 把任务分拆给并行子 Agent

Code with Claude 大会背景：CPO Ami Vora 主导（Mike Krieger 转去主导 Anthropic Labs）。旧金山 05-06 首站，伦敦 05-19，东京 06-10。大会的主题词是「记忆」——Anthropic 认为当前 Agent 最大的瓶颈不是智能，是记忆。

架构分析
Claude Code 泄露文件深度分析：53 个符号工具 + 50 万行符号代码，这是神经符号主义的胜利
Gary Marcus 在 Substack 发文分析 Claude Code 泄露的内部文件。结论：Claude Code 不是「更强的 LLM」，它是一个巨大的符号 AI 系统包裹着一个 LLM——486 个分支点 + 12 层嵌套 IF-THEN + 53 个符号工具 + 50 万行确定性代码。Marcus 称这是「GPT-4 以来 AI 最大进步」，但强调这恰恰不是 pure LLM 的胜利。ACM CACM 同步转载。
泄露文件的具体内容：
核心是一个超大型的条件逻辑系统：486 个分支点，12 层嵌套，完全确定性的符号执行路径。53 个工具的每一个都有明确的调用条件、错误处理、重试逻辑——这些都是硬编码的符号规则，不是 LLM 生成的。

Marcus 的核心论点：
John McCarthy 和 Marvin Minsky（经典 AI 的奠基人）如果看到这个系统，会立刻认出来——这就是他们 60 年前就在做的东西。Claude Code 的「智能」有很大部分来自这 50 万行精心设计的符号代码，而不是 Claude 模型本身。这是一个工程设计的胜利，不是 scaling 的胜利。

对 AI 公司护城河的含义：
如果 Marcus 是对的，那么 AI Coding Agent 的护城河主要来自：① harness 的工程设计（符号规则的积累）；② 领域特定的工具集；③ 对失败模式的系统性处理——而不是「我们用了更强的模型」。这意味着 harness 工程积累是真正可持续的护城河，模型升级无法轻易抹平。
护城河在系统工程设计，而非模型能力。对于垂直 Agent：行业专属的 harness（合规逻辑 + 工具链 + 失败处理）是真正可积累的竞争壁垒。

学术会议
ICLR 2026（里约热内卢，4月22日开幕）：AgentFlow + Flow-GRPO 是当前最重要的长周期 Agent 训练突破
Lambda AI 在 ICLR 2026 发表 12 篇论文，最重要是 AgentFlow：一个可以在任务执行过程中实时训练的 Agent 框架。配套 Flow-GRPO 算法解决了长周期 Agent 的「信用分配难题」（Credit Assignment）。声称实现「7B 参数 Agent 超越 GPT-4o」的效果。另一篇重磅：「Seeing, Listening, Remembering, and Reasoning」——多模态 Agent 长时记忆架构。
信用分配难题（Credit Assignment Problem）：
长周期 Agent 完成一个任务可能需要 100 步操作，但最终结果（成功/失败）只在最后一步才揭晓。如何知道是哪一步操作导致了最终结果？这就是信用分配难题——传统 RL 方法在长序列上效果很差。

Flow-GRPO 的解法：
把长轨迹分解成单步更新，每一步都可以单独评估和优化，然后汇聚成对整体轨迹的优化。关键创新：verifiable trajectory-level reward（可验证的轨迹级奖励），让每个子任务都有明确的正确性标准。

AgentFlow 的架构：
一个「在任务流中训练」的 Agent 系统——Agent 不是先训练再部署，而是在执行任务的过程中持续学习。这与 Anthropic 的 Dreaming（事后回顾）形成对比：AgentFlow 是「边做边学」，Dreaming 是「做完后复盘」。

Anthropic Taught Claude's Agents to Dream. Harvey's Tests Showed a 6x Jump in Completion Rates.
Let's Data Science · 05-11
Anthropic 在 Code with Claude 大会（SF 站，5月6日）发布三个 Managed Agents 功能：Dreaming（跨会话记忆学习）、Outcomes（自评分循环）、Multiagent Orchestration（并行子 Agent）。Harvey 实测：开启 Dreaming 后任务完成率提升 6x。CPO Ami Vora 的定论：当前 Agent 最大的瓶颈是记忆，不是智能。
▸Dreaming 的机制：定时后台进程读取近期会话，提炼三类模式（反复失败、收敛工作流、跨 Agent 偏好），重写记忆库。不修改模型权重，类似海马体记忆巩固。
▸Harvey 案例：法律 Agent 此前每次会话都「忘了」文件格式的特殊处理方式，同样错误反复发生。Dreaming 让这些 workaround 被保留，完成率从基线跳升 6x。
▸三功能状态：Dreaming（研究预览，需申请）、Outcomes（公开 Beta）、Multiagent Orchestration（公开 Beta）。

背景：这场大会为什么重要

Code with Claude 是 Anthropic 的开发者巡回大会，旧金山 5月6日首站，伦敦 5月19日，东京 6月10日。CPO Ami Vora（今年初接替 Mike Krieger）主导。这是 Anthropic 首次以「大会形式」向开发者集中发布功能，模仿 OpenAI DevDay 和 Google I/O 的策略——标志着 Anthropic 的产品化重心从研究发表转向生态建设。

Dreaming 的完整技术细节

Dreaming 不是模型训练。开发者容易把它想成「持续学习」，但这是误解。Dreaming 更像一个「智能日记整理系统」：Agent 在工作时写日记（记录每次操作和结果），Dreaming 在闲置时读日记，提炼出值得长期保留的知识写入正式记忆库。

三类提炼模式：① Recurring mistakes（反复踩的坑）：「每次调用这个 API 都会失败，原因是 X，应该这样处理」；② Workflow convergence（收敛的工作流）：「处理这类法律文档时，Agent 自发地形成了固定的 5 步流程」；③ Cross-agent preferences（跨 Agent 共同偏好）：多个 Agent 都在避免某类操作或偏好某类回答格式。

开发者可以选择：全自动更新记忆（效率高），或需要人类 review 后才更新（更安全）。对于法律、医疗等高风险场景，后者更合适。

Outcomes 和 Multiagent Orchestration

Outcomes 是一个「裁判 Agent」模式：开发者写一个 rubric（评分标准），Outcomes 用一个独立的评估 Agent 对主 Agent 的输出打分，并告诉主 Agent「这里做得不够好，原因是 X」。这给了 Agent 一个结构化的自我改进回路。

Multiagent Orchestration 更直接：主 Agent 收到复杂任务后，分析可以并行执行的子任务，fan-out 给多个子 Agent 同时运行。对于法律文档分析（同时分析 100 份合同）或代码审查（同时检查多个模块），速度可以提升数倍到数十倍。

Harvey 6x 背后的故事

Harvey 是 Anthropic 最重要的企业客户之一（$100M+ 融资的法律 AI 公司），也是 Dreaming 的早期测试伙伴。Harvey 的问题非常典型：法律文档有各种格式特殊性（某类法院文件的日期格式、某家律所的文档模板特殊标注），Agent 每次会话都重新踩相同的坑，因为它「不记得」上一次是怎么处理的。Dreaming 让这些 workaround 被系统性记录并提炼，Agent 在下次遇到相同格式时直接调取经验。

6x 的完成率提升是一个非常大的数字。在企业 AI 软件里，能提升 2-3x 就已经足以驱动购买决策。6x 意味着原本失败率很高的任务类型，现在大部分都能完成。这直接改变了「AI 能做哪些工作」的边界。

对整个 Agent 行业的含义

Ami Vora 的判断值得记住：「Memory lets each agent capture what it learns as it works. Dreaming refines that memory between sessions.」她在说，当前 Agent 的核心瓶颈不是推理能力，而是记忆管理。这与 hwchase17（LangChain 创始人）的「Memory will be the great lock-in」论断完全一致。谁的 Agent 记忆更好、更准确、更易管理，谁就有更强的竞争壁垒。

Your Data Agents Need Context — a16z
a16z · Jason Cui & Jennifer Li · 03-10
a16z 深度分析为什么「数据 Agent」需要一个独立的 Context 层。结论：过去十年现代数据栈的建设把数据集中了，但 Agent 仍然无法有效使用这些数据——因为缺乏「理解数据含义」的上下文。Context 层（知识图谱、语义元数据、业务定义）是 Agent 能否正确推理数据的关键缺失环节。
▸数据栈经历了十年整合（ingestion → transformation → warehouse），但 Agent 在数据上的失败 90% 不是因为数据不够，而是因为 Agent 不理解「这个字段是什么意思」。
▸Context 层的三个组成：① 业务语义（revenue 指的是 GAAP 还是 non-GAAP？）；② 数据血缘（这个数据从哪来、被哪些报表依赖）；③ 使用规范（谁有权看这些数据、看哪个版本）。
▸a16z 的投资判断：Context 层是 AI 时代数据栈的新核心，dbt、Atlan、Alation 等数据目录公司是直接受益者，但真正的机会在「专门为 Agent 设计的 Context Layer 产品」。

问题的本质：Agent 不缺数据，缺理解数据的上下文

一个典型案例：企业数据仓库里有一个字段叫 revenue，但这个 revenue 到底是当天收到的现金、当月确认的收入、还是 ARR？不同部门对这个字段有不同的理解，人类员工通过入职培训和日常沟通会逐渐明白，但 AI Agent 完全不知道。它会理所当然地查询 revenue 然后给出一个数字，但这个数字可能对应着三种完全不同的业务含义。

现代数据栈的三代演进

a16z 梳理了数据架构的演进：第一代，数据孤岛（每个系统自己管数据）；第二代，现代数据栈（集中化 + 清洗 + 可查询，Snowflake/dbt 时代）；现在进入第三代：为 Agent 设计的语义数据层。第二代解决了「数据放在哪里、怎么查询」，第三代要解决「数据是什么意思、Agent 能安全使用哪些数据」。

Context 层的完整图谱

a16z 定义的 Context 层包含四个子层：

  • Semantic Layer（语义层）：业务指标的定义（Monthly Active Users 的精确计算口径）
  • Lineage Graph（血缘图）：数据从哪个系统来、经过哪些 transformation、被哪些报表消费
  • Policy Layer（策略层）：谁有权访问哪些数据、在什么条件下、使用什么版本
  • Knowledge Graph（知识图谱）：不同数据实体之间的关系（客户 → 合同 → 产品 → 收入流）

为什么这对 Agent 特别重要

人类分析师在被老板问到「上个月的 revenue 是多少」时，会根据语境判断：「他是要给 board 看的，应该用 GAAP revenue，不是 bookings。」Agent 没有这个语境判断能力，除非 Context 层明确告诉它「当对话对象是 CFO 时，revenue 指 GAAP non-GAAP-adjusted revenue，排除一次性收益」。

a16z 的投资判断

文章最后，Jason Cui 和 Jennifer Li 直接点出投资逻辑：「每一个正在构建数据 Agent 的公司，都会在 6 个月内遭遇同一堵墙：Context 缺失。」他们预计 Context 层会成为 AI 时代数据基础设施中市值最大的细分赛道之一，因为它是所有数据 Agent 的共同前提依赖——不是某一类产品的差异化功能，而是整个类别的基础设施。

Karpathy at Sequoia AI Ascent 2026：从 Vibe Coding 到 Agentic Engineering
Sequoia AI Ascent · Karpathy · 2026
Karpathy 在 Sequoia AI Ascent 大会接受 Stephanie Zhan（Sequoia 合伙人）专访。他一年前创造了「Vibe Coding」，现在说这不够了——需要「Agentic Engineering」，一种在 AI Agent 协作中保持工程责任的新技能形态。
▸Vibe Coding（个人创作）vs. Agentic Engineering（工程责任）：前者允许你「不关心代码是否存在」，后者要求你在 Agent 出错时能判断出是哪一步出了问题、为什么出错。
▸新技能的核心：不是写代码，而是「监督 unreliable agents 同时不失去对结果的责任」——类似工厂质检而非工人。
▸Karpathy 的判断：「It's all a skill issue.」— 会用 Agent 和不会用 Agent 的人，生产力差距将以量级计。

为什么 Vibe Coding 不够了

Karpathy 一年前提出 Vibe Coding：让 LLM 写代码，你只需要描述你想要什么，不需要理解代码本身。这对个人项目和原型开发非常有效。但专业软件有质量标准——你需要能判断 Agent 的输出是否正确，而不只是看起来正确。

问题在于：Agent 会信心满满地生成看起来正确但实际有严重缺陷的代码。如果你不理解代码（纯 Vibe Coding），你就无法发现这些缺陷。专业软件开发需要的是「能够在不亲自写代码的情况下，仍然对代码质量负责」的能力。

Agentic Engineering 的具体技能

Karpathy 描述了新的技能树：

  • Spec writing（规格书写作）：能够把需求描述得足够精确，让 Agent 理解「什么叫做正确」。PRD 能力成为 coding 能力的前置条件。
  • Agent debugging（Agent 调试）：当 Agent 输出错误时，能够识别是 spec 不清晰、是工具调用失败、是 Agent 幻觉、还是任务超出 Agent 能力范围。
  • Harness design（脚手架设计）：设计 Agent 运行的环境——工具集、约束条件、验证机制——让 Agent 在有限范围内可靠地完成任务。
  • Quality verification（质量验证）：不写代码，但能验证代码正确性——测试用例设计、边界条件识别、安全性审查。

「It's all a skill issue」

Karpathy 的核心论断：差别不在工具，在技能。有人用 Codex 一天完成了过去一周才能完成的工作，有人用同样的工具什么都没做出来。这个差距会随着工具的普及越来越大——当所有人都有相同的 Agent 工具时，只有掌握 Agentic Engineering 技能的人才能真正驾驭它们。

什么是 Agent Labs

swyx 和 Alessio 在 AIE Europe 大会之后录制了这期 debrief，核心发现是一个新的公司形态正在出现：Agent Labs。

传统分类：Model Labs（OpenAI、Anthropic、Google）做模型；Application Companies 调用这些模型做产品。但现在出现了第三类——Agent Labs：在单一垂直领域，同时做模型研究、工程化和商业化。他们不做通用模型，只做「法律 AI 专用模型」或「医疗 AI 专用模型」，但他们确实在做深度的模型工作（post-training、fine-tuning、评估体系）。

编程是第一个 Agent Labs 赛道

编程是第一个被 Agent Labs 完全覆盖的垂直：Cognition（Devin），Anysphere（Cursor），OpenAI 的 Codex 团队（内部 Agent Labs）。这些公司都不只是在调用 GPT-4，它们都在深度定制模型（SWE-bench 专项训练），建设自己的 harness（工具集、审批流程、代码审查循环），并且有清晰的商业模式（订阅或按 token 收费）。

下一批 Agent Labs 赛道

  • 法律：Harvey（$1B+ 融资），Eudia（$100M+ 融资）。已经在做法律专用的模型评估体系（LAB benchmark），构建法律文件 harness。
  • 医疗：Aidoc（本期 $150M），Hippocratic AI（$400M+）。医疗影像、临床决策辅助的专用模型。
  • 金融：Hebbia（$160M），AlphaSense（$1.4B+）。金融研究、报告分析的专用推理。
  • 科学：正在出现，还没有明显赢家。AI for drug discovery 是最热的子方向。

Agent Labs 的核心竞争壁垒

swyx 的关键洞察：Agent Labs 的护城河不是模型参数量，而是「领域专属的 harness + 高质量的领域训练数据 + 深度的客户工作流整合」。这三者缺一不可，且都需要时间积累——不是一夜之间能被通用模型抹平的。

这与 hwchase17（LangChain）的「Memory will be the great lock-in」形成共鸣——Agent Labs 的 harness 本质上是「领域记忆的积累」，谁的 harness 更了解这个领域的工作流，谁就有更强的用户粘性。

No Priors：The Agentic Economy — Jeremy Allaire (Circle CEO)
No Priors · Elad Gil · 2026-04-09
Jeremy Allaire（Circle 联合创始人，USDC 发行方）做客 Elad Gil 的 No Priors，讨论 Agent 经济的金融基础设施。核心观点：AI Agent 已经可以协作，但还不能交易。传统银行根本没有为 Agent 设计，USDC 将成为 Agentic Economy 的原生货币。
▸核心判断：「AI agents can already collaborate, but they can't transact. Traditional banking was not built for agents.」Agent 身份无法通过 KYC，没有信用记录，无法开户，无法签合同。
▸USDC 的优势：可编程稳定币允许 Agent-to-Agent 即时支付，不需要人类审批介入，且每笔交易可验证、可审计、可追溯。
▸Allaire 预测：3-5 年内，USDC 交易量的大部分将来自 Agent-to-Agent 经济活动，而非人对人。Circle 的 Agent Stack 是这个预测的第一个产品化动作。

Agent 为什么需要自己的金融系统

一个具体场景：一个法律 Agent 在处理一个合同审查任务时，需要访问一个专业法律数据库（需要付费订阅）。现在的做法：人类手动给 Agent 提供 API key + 预付费额度。这个过程：① 需要人类介入；② 无法动态调整额度；③ Agent 的消费混入人类的账单，难以追踪；④ 无法自动在预算用完时停止。

如果 Agent 有自己的钱包和支付能力：它可以直接付费、控制自己的预算、在完成任务后向人类报告开支——全程不需要人工干预。这才是真正的自主 Agent。

为什么传统银行不行

传统银行体系的每一个环节都是为人类设计的：KYC（知道你的客户）要求真实姓名和政府 ID，Agent 没有；信用评分依赖历史记录，新的 Agent 没有历史；ACH 转账需要 1-3 个工作日，Agent 的操作是秒级的；开户需要人类签字，Agent 无法签字。

USDC 为 Agent 设计的六个特性

  • Programmability（可编程性）：可以用代码定义支付条件——「只有任务完成才释放付款」
  • Instant settlement（即时结算）：链上交易几秒确认，不需要等 1-3 天
  • Pseudonymity（假名性）：Agent 可以有一个链上地址作为身份，不需要真实 KYC
  • Auditability（可审计性）：所有交易永久记录在链上，满足合规要求
  • Composability（可组合性）：可以把付款逻辑和其他 Agent 操作组合成复杂的业务流程
  • Global（全球可用）：不受银行跨境汇款的限制，Agent 可以向全球任何地址支付

Agentic Economy 的规模预测

Allaire 给出了一个大胆的预测框架：如果全球有 1 亿个 AI Agent，每个 Agent 每天处理 10 笔微支付（均值 $0.1），那就是每天 1 亿 × 10 × $0.1 = $10 亿的 Agent 经济交易量。现在全球每天的支付总量约 $2000 亿，Agent 经济在 3-5 年内可能占到这个数字的 0.5%-5%。对于 Circle 来说，这是「下一个 PayPal 时刻」。
